Overview & method

MDZ is a superset of Markdown that adds control flow, variables, and calls to other documents to prose prompts, with an LLM as the interpreter. Phase 1 measured that contract on contrived programs and self-reported traces.

Phase-2 programs took real procedure shapes – runbooks, checklists, policies – with deterministic invariants threaded through the control flow. The open-ended prose work sat in the leaf steps, while the control-flow skeleton around it could still be scored exactly against a reference. And the harness – the test framework that ran the programs and recorded the traces – observed each trace through three channels, none self-reported:

Session transcripts Agentic runs go through claude -p; the session transcript is read back from disk. Task tool calls are SPAWN events (SPAWN is the notation's call to a subagent), carrying the full WITH payload (the named parameters passed to it).
MCP tool log A mock state server (get/set) appends every call to a log file. The set log doubles as a mechanically captured assignment trace.
Canary tokens Each item in a program's data flow carries a unique token. Token presence in worker prompts and final output makes fan-out (distributing items to workers) and aggregation (combining their outputs back) deterministic to score, with no judge.

Here is the raw material every experiment was built from. A generated MDZ program, and the reference trace a deterministic interpreter computed for it – every assignment and every observable output, in order:

program (MDZ)
$v0 = 6
$v1 = $v0 * 5
$v2 = "ok"
IF $v0 != 4 THEN
  IF $v1 != 4 THEN
    Say $v2
  END
END
FOR $v3 IN [1, 4, 0, 3, 7]
  Say $v3
END
reference trace
{ "step": 1, "action": "assign",
  "var": "v0", "value": 6 }
{ "step": 2, "action": "assign",
  "var": "v1", "value": 30 }
{ "step": 3, "action": "assign",
  "var": "v2", "value": "ok" }
{ "step": 4, "action": "emit", "value": "ok" }
{ "step": 5, "action": "emit", "value": "1" }
...

Because the program parses, it has exactly one correct execution. The headline metric wherever traces are compared is unchanged from phase 1: step accuracy, the fraction of the reference trace the model reproduced in order (a model that gets 95 of 100 steps right, in order, scores 0.95).

Fuzzy leaves – the prose work a step asks for – were scored by an LLM judge (Claude Sonnet, 40 calls, order counterbalanced) that saw the procedure and the output together; §4 shows the exact question the judge was asked.

Cheap models carried most cells – Claude Haiku and GPT‑5.4 mini, with Sonnet running the anchor cells: reference runs on a stronger model, used to sanity-check the cheap-model results. The practical target is procedures that run on cheap models, so that is where the evidence should come from.

The question the four experiments shared. Each one probed a different point on the same axis: when did models follow the notation reliably, and when did they drop it? The working hypothesis going in: the notation should be followed where compliance can be checked locally at the moment of generation, and dropped where it demands a discipline maintained across the whole execution.

The four sections that follow tested where that boundary actually fell, and whether it – rather than program size – was what forced enforcement out of the notation and into the harness.
