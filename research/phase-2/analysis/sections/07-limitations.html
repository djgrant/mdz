Limitations

  • Cell sizes are small. Most cells are n=3–6; E4 ran each task once per model per arm. The headline gaps (0/21 vs 10/10 halts; 18/20 vs 7/20 compliance) survive the noise; the finer rates – haiku's 2-in-12 misrouting, GPT's 1-in-10 wrongly halted clean run – are order-of-magnitude estimates.
  • The internal arm still relied on self-reporting. E3's comparability baseline inherited the phase-1 confound – a run that executes correctly while its self-reported trace omits assignments – and one sonnet run exhibited it. Haiku's 1.0s are trustworthy only because they are perfect; a lower self-reported score would be ambiguous.
  • Store-arm scoring has a formatting component. set values arrived as strings; numeric strings were coerced before comparison, but sonnet's quote-wrapped strings ("\"blue\"") scored as mismatches. This depressed store-arm assign accuracy by a few points; the dominant failure remained the missing calls.
  • One checker was wrong. The hiring-screen forbidden-content regex punished faithful echoes of the procedure (§4). I report the checker’s 18/20 as the headline and the corrected 20/20 in prose; other checkers may harbour subtler versions of the same artefact.
  • The lambda-fidelity probe confounded two things. Making the payload resemble an injection attack tested worker compliance as much as binding fidelity (§2). Binding was measured cleanly – the payloads are in the transcripts; aggregation under that variant was not.
  • USE remains unobservable. USE – the notation's inline skill call – produces no natural event for the harness to record; at best a file read. E2 measured SPAWN; whether USE can be captured and audited is still an open question.
  • Model coverage is narrow. Haiku carried most cells by design; GPT‑5.4 mini appeared in E1 and E4, and sonnet ran a small number of anchor cells in E2 and E3. Cross-family claims (e.g. that misrouting is a small-model behaviour) rest on one model per family.
