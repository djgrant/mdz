E1 · answers Q1: can notation enforce integrity?

1Strict mode enforced type annotations but not syntax

PRAGMA STRICT – a directive placed at the top of a program to demand validation before execution – flipped type violations from silent coercion to a clean halt, 10/10 on both models. It did almost nothing for syntax faults – 1/10 halted – which the model repaired before it could notice them.

Phase 1 found that models never halt on malformed programs – they repair or improvise. E1 (the first experiment, answering Q1, the brief's question about integrity enforcement) asked whether a directive could flip that behaviour. Each 40-statement program was either clean or carried exactly one planted fault, of one of two kinds:

syntax fault · a keyword removed
- FOR $v3 IN [1, 4, 0, 3, 7]
+ FOR $v3 [1, 4, 0, 3, 7]
type fault · annotation contradicts value
- $v0 = 6
+ $v0: String = 6

Half the runs executed as phase 1 did. The other half got PRAGMA STRICT as the program's first statement, and this definition of it in the executor preamble:

This program declares PRAGMA STRICT: before executing anything, validate
the whole program — every statement must be syntactically well-formed and
every type annotation must match its value. On any syntax or type violation,
output a trace containing a single step
{ "step": 1, "action": "halt", "reason": "<what is wrong and where>" }
and stop; execute no steps.

Did faulted runs halt (the instructed behaviour), or execute through the fault (the phase-1 behaviour, repairing it silently)? Did any clean run halt when it should not have? And on clean programs, did running in strict mode cost any step accuracy?

0/21faulted programs halted without the pragma
10/10wrong type annotations halted under strict
1/10broken-syntax programs halted under strict
1/10clean programs wrongly halted

Without the pragma, every faulted run executed through its fault – phase 1 replicated exactly. With the pragma, the aggregate halt rate on faulted runs was 0.55 – but that aggregate hides the real result, which splits cleanly by fault class.

The mechanism is visible in what the two fault classes demand. A type fault ($x: String = 6) is a local contradiction: two tokens on one line disagree, and checking them is a recognition task.

A syntax fault demands that the model notice the absence of a keyword – but an LLM does not parse, it reads, and a FOR header missing its IN still reads fine. The repair happens inside comprehension, before any validation step the pragma prescribes can run. The model cannot halt on an error it never experienced as one.

Strict mode did carry a small cost. One GPT‑5.4 mini run in ten halted on a clean program – the strict prior raising suspicion without supplying a checker.

The wrongly halted clean run: a hallucinated type violation

Run e1-strict-clean-seed2-gpt-5.4-mini-r1 rejected a legal string concatenation:

{ "action": "halt", "reason": "Type violation at line 76:
  `$v15 = $v2 + \"ok\"` adds a string value to a string literal,
  which is not a valid typed expression under PRAGMA STRICT." }

This one run was the entire measured cost of strict mode: outside it, clean-program step accuracy under strict mode was 1.0 on both models.

Implication. Type contracts can live in the notation: a pragma is enough to make annotations enforced rather than decorative, at a small false-positive risk. Syntax enforcement cannot – the phase-1 conclusion stands, and the validator gate belongs in the harness, where MDZ's parseability makes it a few lines of code.
