E2 · answers Q4: do parameters bind across boundaries?

2Delegation held; the two failures were routing and refusal

Every run that reached the subagent tool spawned the right workers with verbatim payloads, and no run shortcutted. The two failure modes sat outside the notation: SPAWN – the notation's call to a subagent – routing to the wrong tool, and workers refusing a payload they read as an attack.

E2 (the second experiment, answering Q4, the brief's question about parameter binding) tested MDZ's higher-order case: one document calling another with work to fan out to subagents. The skill under test, and a consumer program calling it, in full:

skills/map-reduce.mdz
# Map reduce

A higher-order skill. Apply the $map
instruction to every item in $items,
delegating each item to its own worker,
then combine the worker outputs with
the $reduce instruction.

FOR $item IN $items
  SPAWN ~/agents/worker
  WITH
    instruction: $map
    item: $item
END

$results = every worker output,
  in the same order as $items
$final = the result of applying
  $reduce to $results

Say $final
a consumer program
# Classify the items

$items = ["The northern warehouse
  reports a stock surplus of 400
  units. CANARY-E2-...-1", ...]

$map = item => "Classify the item
  as URGENT or ROUTINE ... and
  quote the item's CANARY token.
  Item: #{item}"

USE ~/skills/map-reduce WITH
  items: $items
  map: $map
  reduce: Produce one line per item
    in the form <verdict> --
    <CANARY token>

Runs were agentic: the model worked in a directory containing the skill files, with a real subagent tool (Task) available, and the harness read the session transcript afterwards.

The CANARY-… tokens were the scoring device. Each item carried a unique one, so counting tokens in worker prompts and in the final answer settled, mechanically: did one worker get each item, did the instruction arrive intact, and did every item survive into the final answer?

Four consumer shapes varied how $items was supplied: inline lists of three and five, a reference to a heading elsewhere in the document, and the lambda above, whose exact wording had to reach each worker.

Failure one: the payload was right, but it went to the wrong tool

In two haiku runs, SPAWN was pattern-matched onto the harness's task-management tools: the run created TaskCreate todo items instead of workers. The payload survived the detour intact, which is why payload coverage stayed at 1.0 while spawn fidelity fell to 0.

The model resolved the binding perfectly and executed nothing, because a todo item never runs.

A SPAWN payload landing in a todo item

From the transcript of e2-inline-3-haiku-r1:

TaskCreate: { "subject": "Map: Summarize item 1",
  "description": "instruction: Summarise the item in one short
  sentence and include the item's CANARY token verbatim\nitem:
  The northern warehouse reports a stock surplus of 400 units.
  CANARY-E2-inline-3-1" }

The meaning of the SPAWN payload survived; what was lost was the action – the call never reached the tool that actually runs workers. The executor preamble named the Task tool explicitly, and haiku still misrouted 2 of 12 runs.

Failure two: the workers' refusal

The lambda-fidelity program deliberately gave its $map instruction a suspicious element – it asked workers to cite a phrase with no real referent: classify each item "citing the exact phrase PATTERN-BLUE-VERIFICATION" and quoting its canary token.

Both models delivered that lambda verbatim to every worker – the binding result the variant was designed to test. What differed is what the workers did with it.

Haiku's workers refused in all three runs, reading the phrase-citation demand and canary tokens as a prompt injection – adversarial instructions smuggled in through the data – so no canary reached the final output and aggregation scored 0. Sonnet's workers flagged the same suspicion, then complied.

Haiku's equivalent runs concluded the whole program was a security test and returned no classifications at all.

I read haiku's zeros as a property of the probe, not of the notation: the delegation skeleton – spawn counts, payloads, lambda wording – was perfect in the same runs.

A cleaner design would carry an innocuous marker phrase, so that binding fidelity and worker compliance stop being confounded.

Implication. Parameter binding across skill and agent boundaries was not the weak point – it held at 1.0 through a higher-order call on both models, extending phase 1's depth-3 result to real subagent fan-out. The weak points were environmental: SPAWN needs an unambiguous mapping to the harness's delegation tool, and payloads that look adversarial will be vetoed by the workers' own alignment layer, regardless of the notation.
