E3 · answers Q5: does external state extend execution?

3The state store is treated as documentation, not memory

Models executed the programs correctly and ignored the journalling discipline: the store arm recorded a quarter of the assignments it was instructed to record, while final outputs stayed near-perfect. A negative result, but a different negative than phase 1's ledger.

Phase 1 ruled out a model-maintained prose ledger: it made accuracy worse at twice the cost. E3 replaces it with a real tool. The model executes 100- and 200-statement programs with a state server available, under this instruction:

Use the state store, not your own memory, for program state:
- For EVERY variable assignment, immediately call the state tool
  mcp__state__set with the variable name and its new value.
- Before reading a variable you have not just written, call
  mcp__state__get with the variable name and use the returned value.

The point is accountability as much as capacity. Every tool call lands in a harness-side log, so the set log is the assignment trace – captured mechanically, with nothing self-reported. A comparison arm re-runs the phase-1 setup on the same programs: no tool, state tracked in context, trace self-reported. Two numbers summarise each run. How much of the reference assignment sequence shows up in the recorded assignments, in order (assign accuracy – the set log here, the self-reported trace in the comparison arm), and how much of the program's expected output shows up in the final answer (emit accuracy).

1.0assignments correct when tracked in context (haiku, 8/8 runs)
0.25of the required store writes actually made (mean)
0.97–1.0final outputs correct, both arms
0–4reads from the store per run, against dozens in the program

The gap between those two accuracies is the finding. The model is not failing to compute the state; it is declining to journal it.

The per-run logs show how. A run needing 35 assignments made 15 set calls; one needing 57 made 27; one made 3; one made none at all and still produced a correct output stream. The pattern is one set per variable rather than one per assignment – loop-iteration updates, the bulk of the reference trace, are collapsed into a single write.

The instruction "set after every assignment, get before every read" loses to the model's own working memory: at these sizes its context already tracks state reliably, so the tool round-trip buys it nothing, and the discipline decays into occasional note-taking. The store arm even spent fewer completion tokens at 200 statements (10,177 vs 19,395 for internal) – under-journalling is cheaper than the trace it was meant to replace.

The sonnet anchors sharpen the diagnosis. Sonnet's store runs under-journalled the same way (0.47 and 0.21). And one sonnet internal run scored 0.47 on assigns while its emits were perfect – the phase-1 dropped-assign confound recurring on cue, a correct execution condensed in the self-report. That confound is exactly what the store was meant to eliminate; the store eliminates it only if the model actually journals.

Implication. Voluntary journalling fails for the same reason strict mode fails on syntax: it asks the model to maintain a discipline against its own competence, and competence wins. Two designs follow. Either the ceiling question stops needing the store – the internal arm was perfect at 200 statements (§5) – or state capture moves from instructed to structural: a harness that intercepts assignment-shaped steps, or an executor whose only way to produce a value is through the tool. Instructed compliance is the wrong foundation for accountability.
