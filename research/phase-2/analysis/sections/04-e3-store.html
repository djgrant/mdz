E3 · answers Q5: does external state extend execution?

3The state store is treated as documentation, not memory

Models executed the programs correctly and ignored the journalling discipline: the store arm recorded a quarter of the assignments it was instructed to record, while final outputs stayed near-perfect. A negative result, but with a different cause than phase 1's, where a model-maintained prose ledger degraded accuracy.

Phase 1 ruled out a model-maintained prose ledger: it made accuracy worse at twice the cost. E3 (the third experiment, answering Q5, the brief's question about external state) replaced it with a real tool. The model executed 100- and 200-statement programs with a state server available – a tool exposing set and get calls for writing and reading variables – under this instruction:

Use the state store, not your own memory, for program state:
- For EVERY variable assignment, immediately call the state tool
  mcp__state__set with the variable name and its new value.
- Before reading a variable you have not just written, call
  mcp__state__get with the variable name and use the returned value.

The store serves two purposes: extending capacity, and producing an audit trail. Every tool call landed in a harness-side log, so the set log was the assignment trace – captured mechanically, with nothing self-reported.

Two numbers summarise each run. Assign accuracy is how much of the reference assignment sequence shows up in the recorded assignments, in order (the set log here, the self-reported trace in the comparison arm). Emit accuracy is how much of the program's expected output shows up in the final answer.

1.0assignments correct when tracked in context (haiku, 8/8 runs)
0.25of the required store writes actually made (mean)
0.97–1.0final outputs correct, both arms
0–4reads from the store per run, against dozens in the program

The model was not failing to compute the state; it was declining to record it.

The per-run logs show how. A run needing 35 assignments made 15 set calls; one needing 57 made 27; one made 3; one made none at all and still produced a correct output stream. The pattern was one set per variable rather than one per assignment – loop-iteration updates, the bulk of the reference trace, were collapsed into a single write.

The instruction "set after every assignment, get before every read" lost to the model's own working memory: at these sizes its context already tracked state reliably, so the tool round-trip bought it nothing.

The discipline decayed into occasional note-taking. The store arm even spent fewer completion tokens at 200 statements (10,177 vs 19,395 for internal) – skipping the writes was cheaper than the trace they were meant to replace.

The sonnet anchors sharpened the diagnosis. Sonnet's store runs skipped writes the same way (0.47 and 0.21). And one sonnet internal run scored 0.47 on assigns while its emits were perfect – phase 1 saw the same pattern: a run that executed correctly but whose self-reported trace omitted assignments, so the score understated the execution.

That ambiguity is exactly what the store was meant to eliminate; the store eliminated it only when the model actually made the writes.

Implication. Voluntary recording failed for the same reason strict mode failed on syntax: it asked the model to maintain a discipline that its own competence made unnecessary, and the model defaulted to that competence. Two designs follow. Either the ceiling question stops needing the store – the internal arm was perfect at 200 statements (§5) – or state capture moves from instructed to structural: a harness that intercepts assignment-shaped steps, or an executor whose only way to produce a value is through the tool. Relying on the model to follow a recording instruction is the wrong foundation for an audit trail.
