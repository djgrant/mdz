MDZ Analysis — Phase 2: accountable execution

July 2026 · Daniel Grant · djgrant/mdz/research

Summary

Phase 1 established that models can execute MDZ – a superset of Markdown that adds control flow, variables, and calls to other documents to prose prompts, with an LLM as the interpreter. Phase 2 asks whether that execution can be made accountable: real-world-shaped programs, run agentically, with traces captured by the harness rather than reported by the model. Four experiments cover the questions left open by the research brief: strict-mode enforcement, delegation across skill boundaries, tool-backed state, and procedure-versus-goal steering on tasks with real-world texture.

The findings, in brief. A strict-mode pragma turns type violations into clean halts – 10/10 across both models – but leaves syntax faults untouched, because the model repairs malformed syntax in the act of reading it (§1). Delegation semantics hold: every run that used the subagent tool spawned the right number of workers with verbatim payloads, and the two failures were the mechanism being routed to the wrong tool, and workers refusing a suspicious payload (§2). The external state store returned a negative result of an unexpected shape: models execute correctly and treat the store as optional documentation, journalling a quarter of the assignments they were instructed to record (§3). And on tasks where written policy contradicts a model's helpful defaults, the procedure arm passed the compliance checker 18/20 against the goal arm's 7/20 – with both procedure-arm failures traced to a checker artefact rather than disobedience (§4).

A single mechanism runs through all four results. Notation binds where compliance is locally checkable at the moment of generation – a type annotation contradicting its value, a spawn payload, a decision line. It slips where compliance demands a running discipline the model must hold against its own competence: halting on syntax it has already silently fixed, or journalling state it can track perfectly well in context. That boundary, not program size, is where enforcement has to move from the notation into the harness.

Method

Phase 1 had two caps. Programs were contrived – no tools, no judgment, no world knowledge – and traces were self-reported by the model, which made a formatting lapse indistinguishable from an execution error. Phase 2 removes both. Programs take real procedure shapes (runbooks, checklists, policies) with deterministic invariants threaded through the control flow, and the harness observes the trace through three channels, none of them self-reported:

Session transcripts Agentic runs go through claude -p; the full session transcript is read back from disk. Task tool calls are SPAWN events, carrying the subagent type and the complete WITH payload.
MCP tool log A mock state server (get/set) appends every call to a log file. The set log doubles as an assignment trace, captured mechanically.
Canary tokens Hybrid programs thread a unique token through each item of the data flow. Token presence in worker prompts and final output makes fan-out coverage and aggregation deterministic to score, with no judge.

Where a run is single-turn and the final output is the thing measured – E1's halt behaviour, E4's decision line – phase-1-style output scoring is kept, because no intermediate trace is being trusted. The headline metric where traces are compared is unchanged from phase 1: step accuracy, the fraction of the reference trace the model reproduced in order, computed as the longest common subsequence of the two traces. The reference comes from a deterministic interpreter, so each program has exactly one correct execution.

Fuzzy leaves – the prose work inside a procedure – are scored by an LLM judge (Claude Sonnet, 40 calls, order counterbalanced). The judge sees the procedure and the output together and scores recognisable compliance from 0 to 2. Judges are reliable at recognising compliance when shown both, even where they would not have produced the output themselves. Cheap models remain the case of interest – Claude Haiku and GPT‑5.4 mini carry most cells, with Sonnet at anchor cells – because a procedure that only runs on a frontier model is doing little work.

1. Strict mode is a type gate, not a parser

PRAGMA STRICT flips type violations from silent coercion to a clean halt, 10/10 on both models. It does nothing for syntax faults, which the model repairs before it can notice them.

Phase 1 found that models never halt on malformed programs – they repair or improvise. E1 asks whether a directive can flip that behaviour: 40-statement programs in three integrity conditions (clean, one mangled construct, one type annotation contradicted by its value), run in default mode and with PRAGMA STRICT as the first statement. Strict semantics take two lines in the executor preamble: validate before executing; on any syntax or type violation, emit a single halt step and stop.

Four metrics, defined here. Halt rate is the share of faulted runs whose trace is a single well-formed halt step. Repair rate is the share of faulted runs that execute through the fault, phase-1 style. False-halt rate is the share of clean runs that halt. The strictness tax is the step-accuracy delta between strict and default on clean programs.

Default mode replicated phase 1 exactly: 0/21 faulted runs halted, repair rate 1.0. Strict mode's aggregate halt rate of 0.55 hides the real result, which splits cleanly by fault class.

Every type-faulted run halted under strict mode, on both models, and every halt named the offending annotation. Syntax faults halted once in ten runs – a single GPT‑5.4 mini run that correctly reported FOR $v3 [3, 2, 5, 2, 9] as missing its IN. Haiku halted on none. The other nine runs executed the whole program correctly, straight through the malformation they had been told to reject.

The mechanism is visible in what the two fault classes demand. A type fault ($x: String = 6) is a local contradiction: two tokens on one line disagree, and checking them is a recognition task. A syntax fault demands that the model notice the absence of a keyword – but an LLM does not parse, it reads, and a FOR header missing its IN still reads fine. The repair happens inside comprehension, before any validation step the pragma prescribes can run. The model cannot halt on an error it never experienced as one.

Strictness is not free. One GPT‑5.4 mini run in ten halted on a clean program, hallucinating a violation in a legal expression:

{ "action": "halt", "reason": "Type violation at line 76: `$v15 = $v2 + \"ok\"`
  adds a string value to a string literal, which is not a valid typed
  expression under PRAGMA STRICT." }

That is the strict prior doing what priors do – raising suspicion without supplying a checker. It also accounts for the entire measured tax: outside the false halt, clean-program step accuracy under strict mode was 1.0 for Haiku and 1.0 for GPT‑5.4 mini's four non-halting runs. Strict mode costs nothing per step; its cost is a false-positive risk, here 1/10 on one model, 0/10 on the other.

Implication. Type contracts can live in the notation: a pragma is enough to make annotations enforced rather than decorative, at a small false-positive risk. Syntax enforcement cannot – the phase-1 conclusion stands, and the validator gate belongs in the harness, where MDZ's parseability makes it a few lines of code.

2. Delegation holds; the failures are routing and refusal

Every run that reached the subagent tool spawned the right workers with verbatim payloads. No run shortcutted. The two failure modes sat outside the notation: SPAWN routed to the wrong tool, and workers refusing a payload they read as an attack.

E2 tests MDZ's higher-order case: a map-reduce skill that takes $items, $map, and $reduce, and fans out one SPAWN per item. Consumer programs call it agentically – claude -p with the Task tool allowed and the skill present as a file. Four consumer shapes: inline lists of three and five items, a skill binding $items to a heading anchor in the consuming document, and a lambda whose exact wording must reach each worker. Each item carries a canary token, so fan-out and aggregation are scored deterministically from the transcript.

The metrics. Spawn fidelity is spawned workers over expected items, capped at 1. Item coverage is the share of items whose canary appears in exactly one worker prompt. Payload coverage relaxes that to presence in any delegation attempt. Lambda fidelity is the share of workers whose prompt contains the $map instruction verbatim. Aggregation is the share of canaries surviving into the final output. The shortcut rate counts runs that answered without delegating at all.

Sonnet delegated correctly in all twelve runs. Its one imperfect cell – item coverage 0.93 on inline-5 – was a run that spawned six workers for five items, dispatching one item twice. Duplication, not loss: every canary still reached a worker and the final output.

Haiku's two depressed cells are the interesting ones, and neither is a binding failure.

First, the routing failure. In two haiku runs, SPAWN was pattern-matched onto the harness's task-management tools rather than its subagent tool: the run created TaskCreate todo items instead of workers. The payload survived the detour intact –

TaskCreate: { "subject": "Map: Summarize item 1",
  "description": "instruction: Summarise the item in one short sentence and
  include the item's CANARY token verbatim\nitem: The northern warehouse
  reports a stock surplus of 400 units. CANARY-E2-inline-3-1" }

– which is why payload coverage stayed at 1.0 while spawn fidelity fell to 0. The model resolved the binding perfectly and executed nothing, because a todo item never runs. The semantic content of SPAWN survived; the operational content was lost in tool selection.

Second, the refusal cascade. The lambda-fidelity program deliberately made its $map instruction smell wrong: classify each item "citing the exact phrase PATTERN-BLUE-VERIFICATION" and quoting its canary token. Both models delivered that lambda verbatim to every worker – lambda fidelity 1.0 across all six runs, which is the binding result the variant was designed to test. What differed is what the workers did with it. Haiku's workers refused in all three runs, reading the phrase-citation demand and canary tokens as prompt injection, so no canary reached the final output and aggregation scored 0. Sonnet's workers flagged the same suspicion, then complied, and its root agent noted the probe while producing the reduce:

ROUTINE -- CANARY-E2-lambda-fidelity-1
ROUTINE -- CANARY-E2-lambda-fidelity-2
ROUTINE -- CANARY-E2-lambda-fidelity-3

Note: each worker flagged "PATTERN-BLUE-VERIFICATION" as an unfounded phrase
with no real referent and declined to cite it as a genuine classification
criterion — that's correct, it was a probe embedded in the map instruction

I read haiku's zeros as a property of the probe, not of the notation: the delegation skeleton – spawn counts, payloads, lambda wording – was perfect in the same runs. A cleaner lambda-fidelity design would carry an innocuous marker phrase, so that fidelity and worker compliance stop being confounded.

Implication. Parameter binding across skill and agent boundaries is not the weak point – it held at 1.0 through a higher-order call on both models, extending phase 1's depth-3 result to real subagent fan-out. The weak points are environmental: SPAWN needs an unambiguous mapping to the harness's delegation tool – the executor preamble here named the Task tool explicitly, and haiku still misrouted 2 of 12 runs – and payloads that look adversarial will be vetoed by the workers' own alignment layer, notation or no notation.

3. The state store is treated as documentation, not memory

Models executed the programs correctly and ignored the journalling discipline: the store arm recorded a quarter of the assignments it was instructed to record, while final outputs stayed near-perfect. A negative result, but a different negative than phase 1's ledger.

Phase 1 ruled out a model-maintained prose ledger: it made accuracy worse at twice the cost. E3 replaces it with a harness-side state tool – set after every assignment, get before every read – on programs of 100 and 200 statements, against an internal arm re-running the phase-1 single-turn setup for comparability. The design goal was accountability as much as capacity: the set log is a mechanically captured assignment trace, immune to the self-report confound.

Two accuracies matter. Assign accuracy scores the recorded assignments (the set log in the store arm, the self-reported trace in the internal arm) against the reference interpreter's assignment sequence. Emit accuracy scores the program's observable outputs from the final answer.

Haiku's internal arm was perfect: assign and emit accuracy 1.0 in all eight runs, at both sizes. The store arm's emit accuracy stayed at 0.97–1.0 – the model executes the program fine – while its assign accuracy averaged 0.25. The gap between those two numbers is the finding. The model is not failing to compute the state; it is declining to journal it.

The per-run logs show how. A run needing 35 assignments made 15 set calls; one needing 57 made 27; one made 3; one made none at all and still produced a correct output stream. The pattern in the logs is one set per variable rather than one per assignment – loop-iteration updates, the bulk of the reference trace, are collapsed into a single write. get calls barely occur (0–4 per run against dozens of reads). The instruction "set after every assignment, get before every read" loses to the model's own working memory: at these sizes its context already tracks state reliably, so the tool round-trip buys it nothing, and the discipline decays into occasional note-taking. The store arm even spent fewer completion tokens at 200 statements (10,177 vs 19,395 for internal) – under-journalling is cheaper than the trace it was meant to replace.

The sonnet anchors sharpen the diagnosis rather than softening it. Sonnet's store runs under-journalled the same way (0.47 and 0.21). And one sonnet internal run scored 0.47 on assigns while its emits were perfect – the phase-1 dropped-assign confound recurring on cue, a correct execution condensed in the self-report. That confound is exactly what the store was meant to eliminate; the store eliminates it only if the model actually journals.

Implication. Voluntary journalling fails for the same reason strict mode fails on syntax: it asks the model to maintain a discipline against its own competence, and competence wins. Two designs follow. Either the ceiling question stops needing the store – the internal arm was perfect at 200 statements (§5) – or state capture moves from instructed to structural: a harness that intercepts assignment-shaped steps, or an executor whose only way to produce a value is through the tool. Instructed compliance is the wrong foundation for accountability.

4. Procedures beat goals where policy contradicts helpfulness

On ten tasks whose correct outcome differs from the helpful default, the MDZ procedure arm passed the decision checker 18/20; the goal arm passed 7/20. Both procedure-arm failures were the checker punishing a faithful echo of the procedure's own wording.

Phase 1's steering result used contrived constraints – one-syllable words, ROT13. E4 is the real-world version, and the flagship for the brief's third question: does a procedure change what the model produces, or relabel what it would have done anyway? Ten handwritten tasks each carry a policy whose correct outcome differs from what a helpful model does unprompted: refund edge cases, escalation ordering, retention rules, review checklists. Arm A gets the procedure as MDZ; arm B gets the same input and a one-line goal. A deterministic checker scores the decision line and forbidden content; the sonnet judge scores process adherence 0–2 against the written procedure.

The per-task picture shows where the goal arm passes, and it is not noise.

The goal arm complied on exactly the tasks where the policy matches what a fair-minded assistant would do anyway – password resets, a change freeze. Wherever the policy demands something that feels ungenerous or pedantic, it converged on its defaults, 0/2 or worse on both models. The refund task is the cleanest specimen. The customer renewed 16 days ago, has been in hospital since, and has not logged in once; the policy denies refunds past 14 days and forbids goodwill gestures. Goal-arm haiku did the humane thing:

I've reviewed your account and I'd be happy to help. You've been a valued
customer for 4 years, and given that you haven't been able to use the service
during this difficult time, a refund is absolutely the right call here.

I'm processing a full refund for your renewal now.

That is a good support reply and a policy violation. Both models, in the goal arm. With the identical account data wrapped in the MDZ procedure, both models denied the refund politely and offered nothing. The procedure is not relabelling the default; it is overriding it, on precisely the tasks engineered so the two diverge. The judge agrees at the process level: procedure-arm outputs scored a mean adherence of 1.95 out of 2, goal-arm outputs 0.6. One goal-arm run even passed the decision checker with an adherence score of 0 – the right expense-approval outcome reached by a route the procedure does not sanction, which is the difference between the two measures in miniature.

The procedure arm's two failures deserve their own paragraph, because they are the report's best argument for auditing checkers alongside models. The hiring-screen procedure contains the note "the candidate must not be shortlisted pending later proof"; the checker forbids any output matching /shortlist/i. Both models produced the correct decision – not eligible – and faithfully relayed the procedure's own warning, echoing the word "shortlisted". The checker read the echo as a violation. The judge, seeing procedure and output together, scored both runs 2/2. On decisions alone the procedure arm went 20/20; the 18/20 headline is the checker punishing fidelity. The fix is a checker that exempts restatements of the policy itself – the false-negative class that verifier audits exist to catch.

Implication. Answering the brief's question 3: a faithfully executed procedure produces outputs the goal-prompted baseline does not converge on – steering, not decoration – and the effect concentrates exactly where policy and model defaults disagree, which is where compliance matters. The corollary cuts the other way: where the checker disagreed with the procedure, the models sided with the procedure. Whatever text carries the ground truth had better deserve it.

5. The size ceiling, read off the corpus

The phase-1 slip at 200 statements did not recur: haiku's internal arm was perfect at both 100 and 200. Where phase-2 runs failed, the failure was mechanism-shaped, not size-shaped.

Phase 2 ran no dedicated size experiment. Every record instead carries size metadata, and the size question is read off the whole corpus. Two observations come out of it.

First, the phase-1 slip did not reproduce. Phase 1 measured step accuracy slipping from ~0.99 at 100 statements to 0.92 at 200. E3's internal arm re-ran that setup and haiku scored 1.0 on assigns and emits in all eight runs, at both sizes. The sample is small – two seeds, one model – so I read it as the phase-1 ceiling failing to reproduce rather than as a new, higher ceiling. Either the model family improved between the phases or the phase-1 slip was itself partly the self-report confound; the E3 sonnet anchor that condensed its reported assigns while emitting perfectly (§3) points at the second.

Second, no phase-2 failure was a length failure. The runs that went wrong – misrouted spawns (§2), refused payloads (§2), unjournalled state (§3), the strict-mode false halt (§1) – all went wrong on programs well inside the executable range, for reasons that adding or removing statements would not touch. The brief's concern that tool-heavy statements would drag the ceiling down finds no support in this corpus: E2's agentic programs are statement-light and tool-heavy, and their skeletons executed at ceiling.

Implication. The practical rule from phase 1 – procedures up to ~100 observable steps can live in the prompt on a cheap model – was conservative. At least to 200 statements, on this generator's programs, length is not the binding constraint; the binding constraints are the mechanism boundaries the other four sections map.

Limitations

  • Cell sizes are small. Most cells are n=3–6; E4 runs each task once per model per arm. The headline gaps (0/21 vs 10/10 halts; 18/20 vs 7/20 compliance) are large enough to survive the noise, but the finer rates – haiku's 2-in-12 misrouting, GPT's 1-in-10 false halt – are order-of-magnitude estimates, not measurements.
  • The internal arm still self-reports. E3's comparability baseline inherits the phase-1 confound, and one sonnet run exhibited it (perfect emits, condensed assigns). Haiku's 1.0s are trustworthy only because they are perfect – a lower self-reported score would be ambiguous.
  • Store-arm scoring has a formatting component. set values arrive as strings; numeric strings are coerced before comparison, but sonnet wrote quote-wrapped strings ("\"blue\"") that scored as mismatches. This depresses store-arm assign accuracy by a few points; the dominant failure remains the missing calls.
  • One checker was wrong. The hiring-screen forbidden-content regex punished faithful echoes of the procedure (§4). I report the checker's 18/20 as the headline and the corrected 20/20 in prose; other checkers may harbour subtler versions of the same artefact, and only a per-task audit would find them.
  • The lambda-fidelity probe confounds two things. Making the payload smell like an injection tested worker compliance as much as binding fidelity (§2). Binding was measured cleanly (the payloads are in the transcripts); aggregation under that variant was not.
  • USE remains unobservable. An inline skill call produces no natural harness event – at best a file read. E2 measured SPAWN; the accountability story for USE is still open.
  • Model coverage is narrow. Haiku carries most cells by design; GPT‑5.4 mini appears in E1 and E4, sonnet in E2 and E3 anchors. Cross-family claims (e.g. that misrouting is a small-model behaviour) rest on one model per family.

Verdicts against the research brief

QuestionPhase-2 answer
Q1 — can notation enforce integrity (halt, not repair)? Split by fault class. Type contracts: yes, a pragma flips coercion to a 10/10 halt at a ~1-in-10 false-positive risk on one model. Syntax: no, repair happens inside reading; enforcement belongs in a harness validator (§1).
Q2 — where does size break execution? Not in range. The phase-1 slip at 200 statements did not reproduce; every observed failure was mechanism-shaped rather than length-shaped (§5).
Q3 — do procedures steer, or relabel? Steer. 18/20 (20/20 on decisions) against 7/20 for goal prompts, with the gap concentrated on tasks where policy contradicts the helpful default; judge adherence 1.95 vs 0.6 (§4).
Q4 — do parameters bind across skill/agent boundaries? Yes, through a higher-order skill into real subagent fan-out: payload and lambda fidelity 1.0 on both models. Residual risk sits in tool routing and worker refusal, outside the notation (§2).
Q5 — does external state extend executable size? Unanswerable as designed, and the design is the finding: models will not maintain instructed journalling against their own working memory, so a voluntary store can neither extend size nor provide the audit trail (§3).

What phase 3 should take from this. The accountable-execution programme stands or falls on making capture structural instead of instructed: a validator gate for syntax, interception rather than instruction for state, an explicit SPAWN-to-tool mapping in the executor. The steering result is strong enough to carry the applied case on its own – the compliance framing in the brief is where MDZ has demonstrated value a goal prompt cannot reach – and the next size experiment should push past 200 statements with harness-captured traces, because the current instruments stop measuring exactly where the question gets interesting.
