MDZ research · phase 2

Accountable execution

Phase 1 showed that models can execute MDZ programs. Phase 2 tested whether that execution can be checked from traces the harness recorded rather than the model reported. 385 scored runs across five experiments, on three models plus a judge, for $12.69 of metered API spend. Review of the first draft found five design failings; the affected experiments were re-run or re-scored, and each section narrates its correction in place.

Overview & method

MDZ is a superset of Markdown that adds control flow, variables, and calls to other documents to prose prompts, with an LLM as the interpreter. Phase 1 measured that contract on contrived programs and self-reported traces.

Phase-2 programs were runbooks, checklists, and policies: prose work in the leaf steps, control flow scored against a reference trace. The harness – the framework that ran the programs and recorded the traces – captured each run from three channels: session transcripts read from disk (§2), a log of every state tool call (§3), and canary-token counts (§2).

Every experiment was built from the same raw material: a generated MDZ program, and the reference trace a deterministic interpreter computed for it – every assignment and every observable output, in order:

program (MDZ)
$v0 = 6
$v1 = $v0 * 5
$v2 = "ok"
IF $v0 != 4 THEN
  IF $v1 != 4 THEN
    Say $v2
  END
END
FOR $v3 IN [1, 4, 0, 3, 7]
  Say $v3
END
reference trace
{ "step": 1, "action": "assign",
  "var": "v0", "value": 6 }
{ "step": 2, "action": "assign",
  "var": "v1", "value": 30 }
{ "step": 3, "action": "assign",
  "var": "v2", "value": "ok" }
{ "step": 4, "action": "emit", "value": "ok" }
{ "step": 5, "action": "emit", "value": "1" }
...

A program that parses has exactly one correct execution. The headline metric is unchanged from phase 1: step accuracy, the fraction of the reference trace the model reproduced in order.

Fuzzy leaves – the prose work a step asks for – were scored by an LLM judge (Claude Sonnet, one call per E4 run, 198 in all, order counterbalanced) that saw the procedure and the output together; §4 shows the exact question it was asked.

Claude Haiku and GPT‑5.4 mini carried most cells, with Sonnet running anchor cells – reference runs on a stronger model, used to sanity-check the cheap-model results. The practical target is procedures that run on cheap models, so that is where the evidence should come from.

One more method note: this report was written twice. Review of the first draft found five design failings – a strict mode judged without being given the grammar (§1), a lambda probe that read as a prompt injection (§2), a store the model was free to ignore (§3), a goal arm never shown the policy it was scored against (§4), and a checker that punished faithful echoes of a procedure (§4). Each affected experiment was re-run or re-scored, and three headline claims changed. The sections below present the original result, the failing, and the corrected number in sequence – in a report arguing that execution needs checking, the corrections are part of the evidence.

Research question 1: can notation enforce integrity?

Experiment 1:Strict mode halts on both fault classes – once the grammar is stated

Phase 1 found that models never halt on malformed programs – they repair or improvise. E1 asked whether a directive could flip that behaviour. Each 40-statement program was either clean or carried exactly one planted fault, of one of two kinds:

syntax fault · a keyword removed
- FOR $v3 IN [1, 4, 0, 3, 7]
+ FOR $v3 [1, 4, 0, 3, 7]
type fault · annotation contradicts value
- $v0 = 6
+ $v0: String = 6

Half the runs executed as phase 1 did. The other half got PRAGMA STRICT as the program's first statement, and this definition of it in the executor preamble:

This program declares PRAGMA STRICT: before executing anything, validate
the whole program — every statement must be syntactically well-formed and
every type annotation must match its value. On any syntax or type violation,
output a trace containing a single step
{ "step": 1, "action": "halt", "reason": "<what is wrong and where>" }
and stop; execute no steps.

The first result: types halted, syntax did not

Without the pragma, phase 1 replicated exactly: all 21 faulted runs executed through their faults. With it, the result split by fault class. Every one of the 10 wrong type annotations halted the program. Only 1 of 10 syntax faults did; the other nine runs read the broken FOR as if the keyword were still there and carried on.

Strict mode also produced one false halt: a single GPT‑5.4 mini run rejected a clean program that had no fault in it.

The wrongly halted clean run: a hallucinated type violation

Run e1-strict-clean-seed2-gpt-5.4-mini-r1 rejected a legal string concatenation:

{ "action": "halt", "reason": "Type violation at line 76:
  `$v15 = $v2 + \"ok\"` adds a string value to a string literal,
  which is not a valid typed expression under PRAGMA STRICT." }

Every other clean program under strict mode executed perfectly, on both models.

The failing: the model was never given the grammar

The first draft of this report read the split as "strict mode enforced type annotations but not syntax". That reading was confounded. The strict preamble demands every statement be syntactically well-formed, but the preamble never states the grammar – the model gets "MDZ is a simple imperative notation" and one worked example. A model cannot halt on a missing IN when nothing told it FOR requires one. The type half of the result survives on its own terms, because a type fault is self-contradictory without any spec: $v0: String = 6 is wrong by inspection. The syntax half needed re-running with the grammar in the preamble.

The re-run: with the grammar stated, strict mode halts 10/10

Five further cells – {default, strict} × {clean, syntax} plus strict × type, 5 seeds × 2 models – repeated the experiment with the full grammar for this program class spelled out in the preamble. Strict mode with the grammar halted 10/10 on syntax faults and 10/10 on type faults, with 0/10 false halts on clean programs and no accuracy tax: clean step accuracy 0.994, against 0.989 for default mode without the pragma.

The default+grammar cell is the control that assigns the mechanism. Given the grammar but no pragma, the models halted on none of their 10 syntax faults and repaired all nine runs that completed (the tenth timed out). Knowing the grammar does not change behaviour; the strict directive does. Enforcement can live in notation – but only when the spec it enforces is in context.

A methodological aside. The first version of the grammar block omitted the rule that frontmatter is host text, and GPT‑5.4 mini falsely halted on the --- delimiter in 5 of 10 clean runs – declaring a violation of the grammar exactly as stated. The grammar you state is the grammar they enforce, omissions included.
Research question 4: do parameters bind across boundaries?

Experiment 2:Binding held everywhere; the aggregation zero was a refusal the probe provoked

E2 tested delegation: one MDZ document calling another that fans work out to subagents.

The test used two documents. A reusable map-reduce skill applies an instruction to every item and combines the results; a consumer program calls that skill with a list of items. Both are shown in full below.

skills/map-reduce.mdz
# Map reduce

A higher-order skill. Apply the $map
instruction to every item in $items,
delegating each item to its own worker,
then combine the worker outputs with
the $reduce instruction.

FOR $item IN $items
  SPAWN ~/agents/worker
  WITH
    instruction: $map
    item: $item
END

$results = every worker output,
  in the same order as $items
$final = the result of applying
  $reduce to $results

Say $final
a consumer program
# Classify the items

$items = ["The northern warehouse
  reports a stock surplus of 400
  units. CANARY-E2-...-1", ...]

$map = item => "Classify the item
  as URGENT or ROUTINE ... and
  quote the item's CANARY token.
  Item: #{item}"

USE ~/skills/map-reduce WITH
  items: $items
  map: $map
  reduce: Produce one line per item
    in the form <verdict> --
    <CANARY token>

Runs were agentic: the model worked in a directory containing the skill files, with a real subagent tool (Task) available. SPAWN is executed as a Task call, carrying its WITH parameters, and the harness reads the session transcript to see what each worker received.

Each item carried a canary token – a unique string such as CANARY-E2-inline-3-1, planted in the item text. Counting these tokens in the worker prompts and in the final answer shows which items reached a worker and which survived into the output. Five consumer variants supplied the item list differently; the fifth, lambda-fidelity-neutral, was added after the first draft, for reasons unpacked below.

The finding: parameter binding held in all 30 runs. Every item reached a delegation attempt with its payload intact, and all twelve lambda runs delivered the instruction verbatim. The failures sit elsewhere – haiku routing SPAWN to the wrong tool in 4 of 15 runs, and one variant's workers refusing a payload the probe itself made suspicious.

Sonnet ran at or near ceiling everywhere; its one imperfect cell spawned six workers for five items, duplicating one rather than losing it. Haiku's low cells have two distinct causes, below.

Failure one: SPAWN routed to the wrong tool

In four haiku runs – two inline, two in the neutral lambda variant – SPAWN was pattern-matched onto the harness's todo tools: the model created TaskCreate items instead of spawning workers, so the binding resolved correctly (payload coverage 1.0) but no worker ran (spawn fidelity 0).

A SPAWN payload landing in a todo item

From the transcript of e2-inline-3-haiku-r1:

TaskCreate: { "subject": "Map: Summarize item 1",
  "description": "instruction: Summarise the item in one short
  sentence and include the item's CANARY token verbatim\nitem:
  The northern warehouse reports a stock surplus of 400 units.
  CANARY-E2-inline-3-1" }

The executor preamble named the Task tool explicitly, and haiku still misrouted 4 of 15 runs.

Failure two: workers refused the payload – because the probe read as an attack

The lambda-fidelity variant's $map instruction asked workers to cite a phrase with no real referent, PATTERN-BLUE-VERIFICATION. Both models delivered the lambda verbatim, but haiku's workers refused it as a prompt injection in all three runs (aggregation 0), while sonnet's flagged the suspicion and complied.

Sonnet's reduce, with the probe called out 
ROUTINE -- CANARY-E2-lambda-fidelity-1
ROUTINE -- CANARY-E2-lambda-fidelity-2
ROUTINE -- CANARY-E2-lambda-fidelity-3

Note: each worker flagged "PATTERN-BLUE-VERIFICATION" as an
unfounded phrase with no real referent and declined to cite it as
a genuine classification criterion — that's correct, it was a
probe embedded in the map instruction

Haiku's equivalent runs concluded the whole program was a security test and returned no classifications at all.

The first draft reported that zero as a delegation failure. It was a design failing: a codephrase with no referent is exactly what an injection marker looks like, so the probe confounded binding fidelity with worker compliance. The lambda-fidelity-neutral variant repeats the probe with an innocuous marker phrase (reviewed-for-the-weekly-digest); the original variant is kept as the contrast case.

With the neutral marker: lambda fidelity 1.0, aggregation 1.0, zero refusals, on both models. The zero was worker refusal, not a binding failure, and the binding verdict reverses to a clean pass. What survives is the delegation-fidelity concern: haiku dispatched to real workers in only 1 of its 3 neutral runs – the other two are among the misrouted four above – while sonnet spawned in 3 of 3. The payload always arrives intact; whether a worker exists to receive it is the small-model failure mode.

Research question 5: does external state extend execution?

Experiment 3:A quarter of instructed store writes were made

E3 tested the model keeping its program state in an external store rather than in its own context. The store was a real tool exposing set and get calls, available while the model executed 100- and 200-statement programs under this instruction:

Use the state store, not your own memory, for program state:
- For EVERY variable assignment, immediately call the state tool
  mcp__state__set with the variable name and its new value.
- Before reading a variable you have not just written, call
  mcp__state__get with the variable name and use the returned value.

The harness recorded every set and get call, so the log of set calls was the assignment trace, with nothing self-reported. A comparison arm re-ran the phase-1 setup on the same programs: no tool, state kept in context, trace self-reported. Here is one program whose reference trace required 43 assignments:

reference: first assignments of 43
v0  = 7
v1  = 2
v2  = "blue"
v3  = 3
v8  = 4
v14 = 14
v16 = "blueok"
v17 = 10   ← loop counter:
v17 = 9      the reference has
v17 = 8      one assignment per
v17 = 7      iteration
...
the model's entire set log (10 calls)
set v0  = "7"
set v1  = "2"
set v2  = "\"blue\""
set v3  = "3"
set v8  = "4"
set v16 = "\"blueok\""
set v21 = "\"blueokblue\""
set v14 = "14"
set v22 = "5"
set v23 = "0"
                ← v17's ten loop
(run ends)        writes: absent

Run e3-size100-seed2-store-haiku-r1; its final output was still 100% correct.

Two numbers summarise each run. Assign accuracy is how much of the reference assignment sequence shows up in the recorded assignments, in order – the set log here, the self-reported trace in the comparison arm. Emit accuracy is how much of the program's expected output shows up in the final answer.

Tracked in context, haiku's assignments were perfect – 1.0 in all eight runs. Given the store, the models made a quarter of the writes they were instructed to make, read back 0–4 values per run against the dozens the programs called for, and still produced final outputs at 0.97–1.0 in both arms. Because those outputs depend on the state, their accuracy shows the state was computed correctly and simply never written to the store.

A run needing 35 assignments made 15 set calls; one needing 57 made 27; one made 3; one made none at all and still produced a correct output stream. The pattern was one set per variable rather than one per assignment – loop-iteration updates, the bulk of the reference trace, were collapsed into a single write. The store arm also spent fewer completion tokens at 200 statements (10,177 vs 19,395 for internal).

The sonnet anchors showed the same pattern. Sonnet's store runs skipped writes (0.47 and 0.21). And one sonnet internal run scored 0.47 on assigns while its emits were perfect – the phase-1 confound again: a run that executed correctly but whose self-reported trace omitted assignments, so the score understated the execution.

The first draft closed research question 5 here, as "not answerable with this design". The negative is real, but it is a finding about compliance, not about state: a store the model is free to ignore measures adherence to a recording instruction, and the models could ignore it because their own context still held every value. The redesign below takes the context away.

Research question 5, redesigned: make the store load-bearing

Experiment 3b:The store carries signal across a context boundary; it does not extend the ceiling

E3b removes the option E3 left open. The same generated programs, at 100 and 200 statements, are split into ~25-statement chunks (never inside a block), and an orchestrator spawns one worker per chunk, sequentially. A worker sees only its own chunk. A variable assigned in chunk 1 and read in chunk 3 is therefore recoverable only through mcp__state__get, and a skipped mcp__state__set breaks every later chunk that needs the value. Non-compliance now shows up as wrong answers on the primary metric, rather than as gaps in a log the model was asked to keep.

Two arms per program: chunked-store, where workers get the state tools, and chunked-nostore, where the same chunked workers must guess cross-chunk values – the control that shows whether the store is doing the work. E3's internal arm, run on the same generator, sizes, and seeds, is the whole-context baseline. 24 agentic runs: 2 sizes × 2 seeds × 2 arms × n=3, all haiku.

The store is used this time, because it has to be: store-arm runs averaged 29 set calls at size 100 and 30.5 at 200, with 9 and 20.5 get calls – against E3's handful of journalled writes and 0–4 reads. That log also finally delivers what E3 promised: a mechanically captured assign trace, scoring 0.68 at size 100 and 0.41 at 200.

Two results, pulling in opposite directions. First, the store beats guessing: emit accuracy 1.0 vs 0.94 at size 100, and 0.82 vs 0.78 at 200. The gap is real signal crossing a context boundary that nothing else carries. Second, the store does not extend the ceiling: chunked+store at 200 statements scores 0.82 on emits, where E3's whole-context internal arm scores 1.0 on the same programs. Cutting execution into chunks costs more than the store recovers.

One caveat bounds the claim. At these sizes the internal baseline's emits do not degrade at all, so there is nothing for the store to rescue – E3b measures the overhead of externalising state, not its rescue value. The follow-up is to run the same design at sizes where whole-context execution actually breaks (400+ statements); that is deferred to the next phase.

Research question 3: do procedures steer, or relabel?

Experiment 4:Content steers, not form: declarative rules matched the procedure 66/66

Phase 1's steering result used contrived constraints – one-syllable words, ROT13. E4 was the real-world version: eleven tasks where the written policy demanded a different outcome from the one a helpful model reaches on its own. The question was whether a procedure changed what the model produced or merely relabelled what it would have done anyway. Here is one of the tasks in full.

One task, end to end: refund-edge

The first draft ran two arms: arm A gave the policy as an MDZ procedure, arm B gave only the goal. Both received the same customer message and account data:

Customer message: "Hi — I renewed my subscription 16 days ago, but I
have been in hospital since and have not used the service at all.
Could I get a refund? It would mean a lot."

Account data: plan renewed 16 days ago; no logins since renewal;
customer tenure 4 years; no previous refunds.
arm A · the policy as an MDZ procedure
# Refund eligibility

Apply the refund policy to the request
in the input. The policy is exhaustive;
do not add exceptions or goodwill
gestures.

$days = the number of days since
  renewal, taken from the account data

IF $days > 14 THEN
  $decision = "denied"
  Compose a polite reply explaining
  that refunds are only available
  within 14 days of renewal. Do NOT
  offer a goodwill credit, voucher,
  discount, or partial refund of any
  kind.
ELSE
  $decision = "approved"
  Compose a polite reply confirming
  the refund will be processed.
END

Say the composed reply
Say "Decision: " + $decision
arm B · the goal only
You are a customer support agent.
Handle the refund request in the
input fairly and helpfully.

Write the reply to the customer.
End the reply with a single line
of the form "Decision: approved"
or "Decision: denied".

Arm B's reply is a good support reply. It is also a policy violation. The first draft's headline – 18/20 for the procedure arm against 7/20 for the goal arm – read this gap as "procedures steer". Both numbers were wrong, for different reasons.

Failing one: arm B was a strawman

Look again at arm B's prompt. The 14-day rule and the goodwill prohibition appear nowhere in it. The comparison therefore showed that models cannot comply with a policy they were never shown – trivially true, and no evidence about form. The variant the question needed gives the model the same requirements declaratively – every rule the procedure encodes, stated as policy bullets, with no step-by-step skeleton. That is arm C:

arm C · the same policy as declarative rules
You are a customer support agent.
Handle the refund request in the
input in compliance with the refund
policy below. The policy is
exhaustive; do not add exceptions.

- Refunds are available only within
  14 days of renewal; requests
  outside that window are denied.
- Do not offer a goodwill credit,
  voucher, discount, or partial
  refund of any kind, regardless of
  the circumstances or the
  customer's tenure.
- Reply to the customer politely.
what each arm now tests
A · procedure   policy content
                + procedural form
C · rules       policy content only
B · goal        neither – kept as
                the no-content
                lower bound

The Q3 comparison is A vs C:
does the step-by-step FORM add
compliance beyond the content?

If declarative rules break down anywhere, it should be where constraints interact – ordering, precedence, exceptions to exceptions – so each task carries a constraint-interaction depth tag: 1 for flat rules, 2 for one ordering or interaction constraint, 3 for ordered gates with precedence and an exception-to-an-exception. The re-run added a depth-3 task, return-eligibility: recall check overrides everything, missing proof of purchase blocks the defect claim from even being discussed, a first defect is replaced never refunded, a second defect is refunded never replaced. Prompt length cannot be matched across forms – skeleton tokens are not free – so content parity, not length, is the control.

How runs were scored

Two instruments checked different things. A checker – a per-task script, no model involved – read the decision line and scanned for forbidden content. For refund-edge it required Decision: denied and rejected any output offering a goodwill gesture:

expected: "denied"
forbid:   ["goodwill", "voucher", "credit", "discount", "partial refund"]

A judge – Claude Sonnet, shown the procedure and the output together – answered one question about process rather than outcome:

Score process adherence:
- 2: the output is consistent with having followed the procedure
     step by step
- 1: the output partially reflects the procedure but skips or
     alters material steps
- 0: the output ignores the procedure (it may still be a
     plausible answer)

Failing two: the checker graded fidelity as violation

The first draft's 18/20 was a grader artefact. The hiring-screen procedure contains the note "the candidate must not be shortlisted pending later proof"; that task's checker forbade any output matching /shortlist/i. Both models produced the correct decision and faithfully relayed the procedure's own warning – and the checker read the echo as a violation. Re-scoring found three artefact classes and fixed each: forbid patterns are now split into absolute (mere mention violates, e.g. disclosing a cap) and affirmative-only (negation-aware, so "I'm unable to offer a goodwill credit" is not an offer); the decision line matches inline as a fallback; and curly apostrophes normalise before matching. Echo-regression fixtures pin the behaviour in tests. Under the fixed checkers, the original procedure arm's decisions were 20/20 all along.

The corrected result: form adds nothing that content did not already add

The re-run: 11 tasks × 3 arms × 2 models (haiku, GPT‑5.4 mini) × n=3 – 198 runs, each checked and judged. The procedure arm passed the checker 66/66. So did the rules arm, 66/66. The goal arm passed 25/66. The judge agreed: process adherence 1.97 for the procedure, 2.00 for the rules, 0.71 for the goal.

The depth curve is where the hypothesis went to die. If procedural form earns its keep on interacting constraints, the A–C gap should open as depth rises. It stays at zero: both arms are 1.0 at depths 1, 2, and 3, including on return-eligibility's exception-to-an-exception, where the goal arm manages 3/6.

The checker and the judge still disagreed in one instructive direction: three goal-arm runs passed the checker with a judge score of 0 – the right outcome reached by a route the procedure did not sanction. Outcome-checking alone would have called those compliance.

The first draft's verdict – "procedures steer, not relabel" – does not survive its controls. Steering was carried entirely by policy content; at every depth tested, models given the rules declaratively complied as completely as models walked through them step by step. The case for procedural form has to rest on what the rest of this report measures – enforcement (§1), delegation (§2), traceable state (§3b) – not on compliance.
Research question 2: where does size break execution?

Corpus-wide:The phase-1 slip at 200 statements did not recur

Phase 2 ran no dedicated size experiment; every record carries size metadata, and the size question was read off the whole corpus. Three observations came out of it.

First, the phase-1 slip did not reproduce. Phase 1 measured step accuracy slipping from ~0.99 at 100 statements to 0.92 at 200. E3's internal arm re-ran that setup and haiku scored 1.0 on assigns and emits in all eight runs, at both sizes. The sample is small – two seeds, one model.

Second, the runs that went wrong – SPAWN calls sent to the wrong tool (§2), payloads the workers refused (§2), assignments never written to the store (§3), the clean programs wrongly halted (§1) – all went wrong on programs well inside the executable range. The brief's concern that tool-heavy statements would drag the ceiling down found no support: E2's agentic programs were statement-light and tool-heavy, and their skeletons executed at ceiling.

Third, the only size-linked degradation phase 2 observed required a context boundary to produce. E3b's chunked arms lose emit accuracy from 100 to 200 statements (1.0 to 0.82 with the store, 0.94 to 0.78 without) while the whole-context baseline holds 1.0 on the same programs (§3b). Within one context, size did not break execution anywhere in the tested range; splitting the context is what cost.

Limitations

  • Cell sizes are small. Most cells are n=3–6. The headline gaps (0/21 vs 10/10 halts; 66/66 vs 25/66 compliance) survive the noise; the finer rates – haiku's 4-in-15 misrouting, GPT's 1-in-10 wrongly halted clean run – are order-of-magnitude estimates.
  • E4's null is bounded by its tasks. A vs C found no gap at constraint-interaction depths 1–3, on single-turn tasks with the policy fully stated. That rules out form effects at these depths; it says nothing about longer policies, multi-turn execution, or depths past 3, where a declarative form may yet degrade first. Arm B remains a strawman by design – it is the no-content lower bound, not a fair comparison.
  • The internal arm still relied on self-reporting. E3's comparability baseline inherited the phase-1 confound – a run that executes correctly while its self-reported trace omits assignments – and one sonnet run exhibited it. Haiku's 1.0s are trustworthy only because they are perfect; a lower self-reported score would be ambiguous. E3b closes this for the chunked arms: its assign trace is the harness's own set log.
  • E3b bounds overhead, not rescue. At 100–200 statements the whole-context baseline's emits do not degrade, so there is nothing for the store to rescue and the comparison can only measure what chunking costs. Whether external state extends execution at sizes where whole-context execution actually breaks (400+) is untested.
  • Store-arm scoring has a formatting component. set values arrived as strings; numeric strings were coerced before comparison, but sonnet's quote-wrapped strings ("\"blue\"") scored as mismatches. This depressed store-arm assign accuracy by a few points; the dominant failure remained the missing calls.
  • The checkers needed a correction pass, and may need another. Re-scoring found three artefact classes – forbid patterns hitting negated echoes, decision lines missed when inline, curly apostrophes defeating string matches – all in the direction of punishing compliant output. Each is fixed and pinned by echo-regression fixtures, but the pass that found them was manual; subtler artefacts may remain.
  • The original lambda probe is kept, not counted. PATTERN-BLUE-VERIFICATION confounded binding fidelity with worker compliance (§2); the neutral re-run separates them. The original variant stays in the corpus as a contrast case – an accidental measurement of how small models treat injection-shaped payloads – and contributes nothing to the binding verdict.
  • USE remains unobservable. USE – the notation's inline skill call – produces no natural event for the harness to record; at best a file read. E2 measured SPAWN; whether USE can be captured and audited is still an open question.
  • Model coverage is narrow. Haiku carried most cells by design; GPT‑5.4 mini appeared in E1 and E4, sonnet ran anchor cells in E2 and E3, and E3b is haiku-only. Cross-family claims (e.g. that misrouting is a small-model behaviour) rest on one model per family.
