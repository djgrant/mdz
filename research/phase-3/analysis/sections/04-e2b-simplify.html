Does diverge-then-select beat a single context? Subjective selection

Experiment 2b:Sonnet did not run the program

E2b swaps the benchmark for taste. Targets are over-abstracted modules (~300–400 lines, passing test suites) whose abstraction debt is layered — indirection wrapping duplication wrapping speculative generality — so three successive rounds of genuine simplification are each productive. /simplify is the compound MDZ skill: three directional heuristics fan out per iteration, an inner reduce picks the winner, the winner seeds the next iteration, and a spawned fresh-context judge picks the best iteration at the end. /simplify-goal holds the same heuristics and plan in prose; /simplify-ralph is the same heuristics as a single-pass instruction run three times, fresh sessions, last pass ships unjudged. 2 targets × 3 variants × 2 models × n=3 = 36 runs, scored by a pairwise sonnet judge — order counterbalanced, both candidates presented with the original — across the three arm pairings per target/model/rep: 36 verdicts.

Finding 1: the program was not executed by the stronger model. Five of six sonnet skill runs spawned iteration 1's three workers and then did everything else in the coordinator's own context; the sixth fanned out fully but, like every sonnet run, never spawned the final judge. From iteration 2 onward there were no fresh contexts, no reduce workers, and no selection — one long session improvising under the skill's name. The headline judge tallies (ralph 9–3 over the skill, goal 7–5 over it) therefore compare ralph against a program that was never run; they are not evidence about diverge-then-select. The model split matters: haiku ran the shape almost faithfully (full fan-out 5/6, judge 4/6), and in E2a — cheap one-line strategy payloads — sonnet complied too. The defection is specific to expensive spawns: nine 400-line-file relays that look redundant to a coordinator that believes it already has the answer. Phase 2's delegation-fidelity claim is hereby scoped to cheap payloads; notation-level orchestration is not self-enforcing at real workload sizes.

Finding 2: where the program did run, it lost on behaviour drift, not simplification quality. Within haiku's compliant runs the skill went 1–4 against ralph, and the judge's reasons cluster: the skill's shipped code had silently dropped behaviour — resolveOptions/defaults gone in one run, locale support gone in another — that the test suite did not cover. The cause is in the program text: the map lambda passes the module by value (Code: {$code}), so a compliant run performs ~9 whole-file text regenerations plus 2 selections, the reducer compares candidates only to each other, and each iteration seeds from the previous iteration's text with no path back to the original file. Loss compounds and nothing in the loop can detect it. Ralph makes three in-place edits with the original repo state always readable; it drifted once, the skill drifted repeatedly. The skill's one win was the run where it preserved semantics exactly — and there it beat ralph.

Verdict: E2b returns two mechanism findings and no verdict on diverge-then-select. (1) Sonnet abandoned the program when its spawns became expensive — orchestration written in notation was followed by the weaker model and overridden by the stronger one. (2) In the compliant subset the skill lost because passing code by value through a relay of contexts leaks behaviour that neither the reducer nor the tests can see. A by-reference variant — workers editing the file in the sandbox, anchored to the original — is required before the shape itself can be judged.
